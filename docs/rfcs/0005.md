## Проблема

После реализации RFC-0002 — RFC-0004 в CLI доступны команды для создания, просмотра, редактирования, управления статусом и валидации RFC-документов. Однако поле `links` во frontmatter до сих пор заполняется только вручную — нет CLI-команд для управления связями между RFC и файлами исходного кода.

Это приводит к следующим проблемам:

1. **Отсутствие трассировки RFC → код.** Разработчик не может быстро узнать, какие файлы реализуют конкретный RFC, и наоборот — какой RFC описывает данный модуль. Поле `links` остаётся пустым или заполняется вручную с ошибками.

2. **Нет навигации по зависимостям.** Поле `dependencies` содержит номера RFC-зависимостей, но нет команды для визуализации дерева зависимостей. Разработчик вынужден открывать каждый RFC вручную, чтобы понять граф зависимостей.

3. **`check` не может полноценно валидировать `links`.** Команда `check` из RFC-0004 проверяет, что файлы из `links` существуют, но без CLI-команд для управления ссылками поле часто пустое — проверка бессмысленна.

4. **Будущая команда `doctor` (RFC-0001) зависит от наличия данных в `links` и `dependencies`.** Без `link`/`unlink`/`deps` реализация `doctor` невозможна.

Эти три команды замыкают цикл: RFC-документы становятся не изолированными текстовыми файлами, а узлами графа, связанного с кодовой базой и друг с другом.

## Задача

Реализовать команды `link`, `unlink` и `deps` — инструменты для управления связями RFC с исходным кодом и навигации по зависимостям.

### Требования к `link`

1. Команда принимает номер RFC и путь к файлу: `rfc-cli link <number> <path>`.
2. Путь проверяется на существование относительно корня проекта.
3. Путь нормализуется и сохраняется относительно корня проекта (без ведущего `./`).
4. Если путь уже присутствует в `links` — вывести сообщение и завершиться без ошибки: `RFC-NNNN: link already exists: <path>`.
5. Команда добавляет путь в поле `links` во frontmatter RFC-файла.
6. Команда обновляет запись в индексе `.index.json`.
7. Для `accepted`/`implemented` RFC — блокировать добавление ссылки, если не передан `--force`. При `--force` — обновить `content_hash` после изменения.
8. Вывести подтверждение: `RFC-NNNN: linked <path> ✅`.

### Требования к `unlink`

1. Команда принимает номер RFC и путь: `rfc-cli unlink <number> <path>`.
2. Если путь отсутствует в `links` — вернуть ошибку: `RFC-NNNN: link not found: <path>`.
3. Команда удаляет путь из поля `links` во frontmatter RFC-файла.
4. Команда обновляет запись в индексе `.index.json`.
5. Для `accepted`/`implemented` RFC — блокировать удаление ссылки, если не передан `--force`. При `--force` — обновить `content_hash` после изменения.
6. Вывести подтверждение: `RFC-NNNN: unlinked <path> ✅`.

### Требования к `deps`

1. Команда принимает номер RFC: `rfc-cli deps <number>`.
2. Читает поле `dependencies` из индекса.
3. Для каждой зависимости подтягивает `title` и `status` из индекса.
4. Выводит дерево зависимостей в формате:
   ```
   RFC-0007 depends on:
     - RFC-0003 (config) [implemented]
     - RFC-0005 (logging) [accepted]
   ```
5. Если зависимостей нет — выводит: `RFC-NNNN has no dependencies.`
6. Если зависимость ссылается на несуществующий RFC — выводит предупреждение: `  - RFC-NNNN (not found) ⚠️`.
7. Опционально: флаг `--reverse` для показа обратных зависимостей (кто зависит от данного RFC).

### Расширение rfclib/rfc.rs

Для `link` и `unlink` нужна функция обновления спискового поля `links` во frontmatter. Текущая `update_frontmatter_field` работает со скалярными значениями. Добавим две новые функции:

```rust
/// Adds an item to a list field in YAML frontmatter.
/// Returns the full file content after the update.
pub fn add_to_frontmatter_list(
    file_path: &Path,
    field: &str,
    value: &str,
) -> Result<String, String>
```

```rust
/// Removes an item from a list field in YAML frontmatter.
/// Returns the full file content after the update.
pub fn remove_from_frontmatter_list(
    file_path: &Path,
    field: &str,
    value: &str,
) -> Result<String, String>
```

Стратегия обновления frontmatter для списков:

- Поле `links` может быть в двух YAML-форматах:
  - Inline: `links: [src/main.rs, src/cli.rs]`
  - Block:
    ```
    links:
      - src/main.rs
      - src/cli.rs
    ```
- Для простоты и единообразия при записи всегда используется block-формат.
- При чтении поддерживаются оба формата (через `parse_frontmatter`, который уже использует `serde_yaml`).
- Пустой список записывается как `links: []`.

Алгоритм `add_to_frontmatter_list`:

1. Прочитать файл и найти границы frontmatter (`---` ... `---`).
2. Распарсить frontmatter через `parse_frontmatter`, получить текущий список.
3. Проверить, что значение ещё не в списке.
4. Добавить значение в список.
5. Перестроить блок frontmatter: заменить поле `links` (или `dependencies`) на обновлённый block-формат.
6. Записать файл.

Алгоритм `remove_from_frontmatter_list`:

1. Аналогичен `add_to_frontmatter_list`, но удаляет значение из списка.
2. Если список стал пустым — записать `links: []`.

### Нормализация путей

Функция `normalize_link_path` обеспечивает единообразие путей в `links`:

```rust
/// Normalizes a link path: removes leading `./`, ensures it's relative.
pub fn normalize_link_path(path: &str) -> String
```

- Убирает ведущий `./`.
- Заменяет `\` на `/` (для совместимости с Windows-путями в репозитории).
- Не допускает абсолютные пути (начинающиеся с `/`) — возвращает ошибку.

### Команда link (commands/link.rs)

```rust
pub fn execute(
    project_root: &Path,
    number: &str,
    link_path: &str,
    force: bool,
) -> Result<(), String>
```

Алгоритм:

1. Нормализовать номер RFC (`normalize_number`).
2. Проверить, что файл RFC существует.
3. Нормализовать путь (`normalize_link_path`).
4. Проверить, что целевой файл существует на диске: `project_root.join(&normalized_path).exists()`.
5. Загрузить и обновить индекс (`load_index`, `refresh_index`).
6. Проверить статус RFC — если `accepted`/`implemented` и нет `--force`, вернуть ошибку.
7. Прочитать frontmatter и проверить, нет ли уже такого пути в `links`.
8. Если путь уже есть — вывести сообщение и выйти (не ошибка).
9. Вызвать `add_to_frontmatter_list` для добавления пути.
10. Обновить индекс: перечитать frontmatter, обновить `mtime`, пересчитать `content_hash` если `accepted`/`implemented` и `--force`.
11. Вывести подтверждение.

### Команда deps (commands/deps.rs)

```rust
pub fn execute(
    project_root: &Path,
    number: &str,
    reverse: bool,
) -> Result<(), String>
```

Алгоритм прямых зависимостей (`--reverse` не указан):

1. Нормализовать номер RFC.
2. Загрузить и обновить индекс.
3. Найти запись в индексе по номеру.
4. Прочитать `dependencies`.
5. Для каждой зависимости извлечь номер (из формата `RFC-NNNN`).
6. Найти в индексе соответствующую запись — получить `title` и `status`.
7. Вывести дерево.

Алгоритм обратных зависимостей (`--reverse`):

1. Нормализовать номер RFC и сформировать строку `RFC-NNNN`.
2. Загрузить и обновить индекс.
3. Пройтись по всем записям индекса.
4. Отобрать те, у которых `dependencies` содержит `RFC-NNNN`.
5. Вывести в формате:
   ```
   RFC-0001 is depended on by:
     - RFC-0002 (реализация команд init и new) [implemented]
     - RFC-0003 (реализация команд list, view, status и edit) [implemented]
   ```
6. Если обратных зависимостей нет — вывести: `RFC-NNNN has no reverse dependencies.`

### Взаимодействие с существующими командами

- **`check`**: уже проверяет, что файлы из `links` существуют. После реализации `link`/`unlink` эта проверка становится полезной — ссылки будут реально заполняться.
- **`edit`/`set`**: при изменении frontmatter через `edit` или `set` индекс обновляется. `link`/`unlink` используют тот же паттерн обновления.
- **`reindex`**: при пересборке индекса `links` берутся из frontmatter — `link`/`unlink` пишут именно туда, поэтому `reindex` работает корректно.

## Альтернативы

### 1. Хранить ссылки только в индексе, не во frontmatter

Ссылки можно было бы хранить только в `.index.json`, не трогая файл RFC. Это упростило бы реализацию — не нужно модифицировать frontmatter.

**Отклонено**: противоречит RFC-0001, где `links` — поле frontmatter. Индекс — кэш, воспроизводимый из файлов. Если ссылки только в индексе, `reindex` их потеряет.

### 2. Автоматическое определение связей по `grep`

Вместо ручного `link` — сканировать код и автоматически находить упоминания `RFC-NNNN` в комментариях.

**Отклонено на данном этапе**: это задача для `doctor`, а не для `link`. Автоматика — дополнение к явным ссылкам, не замена. Можно добавить позже как часть `doctor`.

### 3. `deps` без `--reverse`

Можно не реализовывать обратные зависимости и оставить только прямые.

**Отклонено**: обратные зависимости — ключевая информация при принятии решения о supersede/deprecate. Реализация тривиальна (проход по индексу), нет причин откладывать.

### 4. `link` / `unlink` без проверки `--force` для accepted/implemented

Можно разрешить свободное добавление/удаление ссылок для любого статуса, считая это метаданными, а не содержимым.

**Отклонено**: изменение `links` меняет frontmatter → меняет файл → ломает `content_hash`. Для консистентности с `edit` используем тот же паттерн `--force` + пересчёт хэша.
