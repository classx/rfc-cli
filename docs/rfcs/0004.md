## Problem

После реализации RFC-0003 (`list`, `view`, `status`, `edit`) в проекте можно просматривать и редактировать RFC, но нет возможности:

1. **Менять статус RFC через CLI.** Переходы `draft → review → accepted → implemented` и другие выполняются вручную — пользователь открывает файл, меняет поле `status`, и при этом не фиксируется `content_hash`, не валидируются допустимые переходы, не обновляется индекс атомарно. Это нарушает гарантии целостности из RFC-0001.

2. **Валидировать RFC-документы.** Нет способа проверить, что frontmatter корректен, обязательные секции присутствуют, зависимости существуют, а файлы в `links` не удалены. Ошибки обнаруживаются только случайно.

3. **Восстановить повреждённый индекс.** Если `.index.json` повреждён, рассинхронизирован или удалён — единственный способ восстановить его — удалить и пересоздать вручную. `refresh_index` обновляет только изменённые записи по `mtime`, но не пересобирает индекс с нуля.

Эти три команды — ядро управления жизненным циклом RFC. Без `set` процесс переходов неуправляем, без `check` — нет автоматической валидации, без `reindex` — нет страховки от повреждения индекса.

## Goal

Реализовать команды `set`, `check` и `reindex` — набор для управления жизненным циклом и целостностью RFC-документов.

### Requirements for `set`

1. Команда должна принимать номер RFC и целевой статус: `rfc-cli set <number> <status>`.
2. Команда должна валидировать переход из текущего статуса в целевой по таблице допустимых переходов (RFC-0001).
3. Допустимые переходы:
   - `draft → review`
   - `draft → deprecated`
   - `review → accepted`
   - `review → draft` (возврат на доработку)
   - `accepted → implemented`
   - `accepted → deprecated`
   - `accepted → superseded`
   - `implemented → superseded`
   - `implemented → deprecated`
4. При недопустимом переходе — вернуть ошибку: `Error: transition <from> → <to> is not allowed.`
5. При переходе в `superseded` команда должна требовать флаг `--by <number>`: `rfc-cli set <number> superseded --by <superseding_number>`.
6. При переходе в `superseded` команда должна проверить, что замещающий RFC существует. Если не существует — вернуть ошибку.
7. При переходе в `superseded` команда должна записать номер замещающего RFC в поле `superseded_by` во frontmatter.
8. При переходе в `accepted` или `implemented` — команда должна вычислить SHA-256 содержимого файла и сохранить `content_hash` в индексе.
9. Команда должна атомарно обновить: поле `status` во frontmatter файла, запись в `.index.json` (статус, mtime, content_hash, superseded_by).
10. После успешного перехода — вывести: `RFC-NNNN (title): <from> → <to> ✅`
11. При переходе в `superseded` — вывести: `RFC-NNNN (title): <from> → superseded (by RFC-MMMM) ✅`
12. Номер RFC нормализуется (как в RFC-0003): `rfc-cli set 1 review` эквивалентно `rfc-cli set 0001 review`.

### Requirements for `check`

13. Команда может принимать номер RFC опционально: `rfc-cli check [<number>]`.
14. Без аргумента — проверяет все RFC в `docs/rfcs/`.
15. С аргументом — проверяет только указанный RFC.
16. Проверки:
    - a) YAML frontmatter парсится без ошибок.
    - b) Все обязательные поля frontmatter присутствуют: `title`, `status`, `dependencies`, `superseded_by`, `links`.
    - c) Значение `status` входит в список допустимых: `draft`, `review`, `accepted`, `implemented`, `superseded`, `deprecated`.
    - d) Обязательные секции Markdown присутствуют: `## Проблема`, `## Задача`, `## Дизайн`, `## Альтернативы`.
    - e) Номер в имени файла совпадает с номером в `title` (файл `0003.md` → title содержит `RFC-0003`).
    - f) Зависимости ссылаются на существующие RFC-файлы.
    - g) Файлы в `links` существуют в проекте (относительно корня проекта).
    - h) Для RFC в статусах `accepted`/`implemented` — `content_hash` в индексе совпадает с фактическим SHA-256 файла.
17. Результат каждой проверки: список ошибок с номером RFC и описанием, или `All checks passed. ✅` если ошибок нет.
18. Код выхода: `0` если все проверки пройдены, `1` если есть ошибки.
19. После проверки команда должна выполнить `refresh_index` для актуализации индекса.

### Requirements for `reindex`

20. Команда не принимает аргументов: `rfc-cli reindex`.
21. Команда должна полностью перестроить `.index.json` с нуля: сканировать все `docs/rfcs/*.md`, парсить frontmatter каждого, создать новый индекс.
22. Для RFC в статусах `accepted`/`implemented` — вычислить и сохранить `content_hash`.
23. Для остальных статусов — `content_hash` должен быть `null`.
24. После завершения — вывести: `Reindexed N RFCs.`
25. Если `docs/rfcs/` не существует — вернуть ошибку с предложением выполнить `rfc-cli init`.

### Extending CLI (cli.rs)

```rust
#[derive(Subcommand)]
pub enum Commands {
    // ... существующие Init, New, List, View, Status, Edit ...

    /// Изменить статус RFC
    Set {
        /// Номер RFC
        number: String,
        /// Целевой статус
        status: String,
        /// Замещающий RFC (для перехода в superseded)
        #[arg(long)]
        by: Option<String>,
    },
    /// Проверить валидность RFC
    Check {
        /// Номер RFC (если не указан — проверить все)

    },
    /// Полностью перестроить индекс
    Reindex,
}
```

### Valid Transitions Table

Размещается в `src/rfclib/rfc.rs`:

```rust
/// Допустимые статусы
pub const VALID_STATUSES: &[&str] = &[
    "draft", "review", "accepted", "implemented", "superseded", "deprecated",
];

/// Проверяет, допустим ли переход из одного статуса в другой
pub fn is_valid_transition(from: &str, to: &str) -> bool {
    matches!(
        (from, to),
        ("draft", "review")
            | ("draft", "deprecated")
            | ("review", "accepted")
            | ("review", "draft")
            | ("accepted", "implemented")
            | ("accepted", "deprecated")
            | ("accepted", "superseded")
            | ("implemented", "superseded")
            | ("implemented", "deprecated")
    )
}
```

### Updating Frontmatter in File

Размещается в `src/rfclib/rfc.rs`:

```rust
/// Заменяет значение поля в YAML frontmatter файла.
/// Перезаписывает файл с обновлённым frontmatter, сохраняя остальной контент.
pub fn update_frontmatter_field(
    file_path: &Path,
    field: &str,
    value: &str,
) -> Result<String, String>
```

Алгоритм:
1. Прочитать файл
2. Найти границы frontmatter (между `---` маркерами)
3. Заменить строку с полем `field` на `field: value`
4. Для `superseded_by` — заменить `null` на значение
5. Записать файл
6. Вернуть полное содержимое файла после записи (для вычисления content_hash)

### Command set (commands/set.rs)

```rust
pub fn execute(
    project_root: &Path,
    number: &str,
    target_status: &str,
    by: Option<&str>,
) -> Result<(), String>
```

Алгоритм:
1. Нормализовать номер, определить путь к файлу
2. Проверить, что файл существует
3. Загрузить и актуализировать индекс (`load_index` + `refresh_index`)
4. Найти запись в индексе, получить текущий статус
5. Проверить, что `target_status` входит в `VALID_STATUSES`
6. Проверить, что переход допустим (`is_valid_transition`)
7. Если `target_status == "superseded"`:
   - Проверить, что `--by` указан. Если нет — ошибка: `Error: --by <number> is required for transition to superseded.`
   - Нормализовать номер замещающего RFC
   - Проверить, что замещающий RFC существует. Если нет — ошибка: `Error: RFC-MMMM not found.`
   - Обновить поле `superseded_by` в frontmatter: `update_frontmatter_field(path, "superseded_by", "RFC-MMMM")`
8. Обновить поле `status` в frontmatter: `update_frontmatter_field(path, "status", target_status)`
9. Если `target_status` == `accepted` или `implemented`:
   - Прочитать содержимое файла
   - Вычислить `content_hash` (SHA-256)
10. Обновить запись в индексе: status, mtime, content_hash, superseded_by
11. Сохранить индекс
12. Вывести сообщение об успешном переходе

### Command check (commands/check.rs)

```rust
pub fn execute(project_root: &Path, number: Option<&str>) -> Result<(), String>
```

Алгоритм:
1. Определить список файлов для проверки:
   - Если `number` задан — один файл
   - Иначе — все `docs/rfcs/*.md`
2. Загрузить и актуализировать индекс
3. Для каждого файла выполнить проверки (a)–(h):
   - a) Попытаться парсить frontmatter. При ошибке — записать в список ошибок
   - b) Проверить наличие обязательных полей (парсинг в `RfcFrontmatter` уже гарантирует это через serde, но проверить, что значения не пустые)
   - c) Проверить `status ∈ VALID_STATUSES`
   - d) Проверить наличие обязательных секций: поиск строк `## Проблема`, `## Задача`, `## Дизайн`, `## Альтернативы` в содержимом файла
   - e) Извлечь номер из имени файла, проверить что `title` содержит `RFC-NNNN`
   - f) Для каждого элемента `dependencies` — проверить, что соответствующий файл существует (извлечь номер из `RFC-NNNN`, построить путь)
   - g) Для каждого элемента `links` — проверить, что файл существует относительно `project_root`
   - h) Если статус `accepted`/`implemented` и в индексе есть `content_hash` — вычислить SHA-256 файла и сравнить
4. Собрать все ошибки
5. Вывести результат:
   - Если ошибок нет: `All checks passed. ✅`
   - Если есть ошибки — вывести каждую в формате: `RFC-NNNN: <описание ошибки>`
6. Выполнить `refresh_index`
7. Код выхода: `0` если нет ошибок, `1` если есть

Формат вывода ошибок:

```
RFC-0003: invalid status 'draf' (expected one of: draft, review, accepted, implemented, superseded, deprecated)
RFC-0003: missing required section '## Альтернативы'
RFC-0005: dependency RFC-0099 not found
RFC-0005: dead link: src/old_module.rs (file not found)
RFC-0002: content_hash mismatch (accepted RFC was modified)

Found 5 errors in 2 RFCs.
```

### Command reindex (commands/reindex.rs)

```rust
pub fn execute(project_root: &Path) -> Result<(), String>
```

Алгоритм:
1. Проверить, что `docs/rfcs/` существует. Если нет — ошибка: `Error: docs/rfcs/ not found. Run "rfc-cli init" first.`
2. Сканировать все `docs/rfcs/*.md`
3. Для каждого файла:
   - Прочитать содержимое
   - Парсить frontmatter
   - Получить mtime из метаданных файла
   - Если статус `accepted`/`implemented` — вычислить `content_hash` (SHA-256)
   - Иначе — `content_hash = null`
   - Создать `IndexEntry`
4. Отсортировать записи по номеру
5. Записать новый `.index.json` (полностью заменяет старый)
6. Вывести: `Reindexed N RFCs.`

## Alternatives

### 1. Merge `check` and `reindex`

Сделать `reindex` частью `check`: `rfc-cli check --reindex`.

**Почему не подходит:** разные сценарии использования. `check` — это валидация (read-only, может упасть с ошибкой). `reindex` — это восстановление (write, всегда успешно). Объединение усложнит семантику: пользователь хочет «просто проверить» без побочных эффектов, или «просто пересобрать индекс» без проверки формата.

### 2. Automatic `set` when editing frontmatter

Определять изменение статуса в `edit` и автоматически вызывать логику `set`.

**Почему не подходит:** `edit` открывает произвольный редактор — нет контроля, что именно изменилось. Определение diff frontmatter до/после редактирования добавляет сложность. `set` как явная команда прозрачнее и предсказуемее.

### 3. `set` without transition validation

Разрешить любые переходы статусов, оставив ответственность пользователю.

**Почему не подходит:** весь смысл RFC-процесса — в дисциплине. Если можно перевести `implemented → draft`, теряется гарантия, что принятый RFC не откатят без следа. Таблица переходов зафиксирована в RFC-0001 (`accepted`).

### 4. `check` as a pre-commit hook

Запускать `check` автоматически при коммите вместо явной команды.

**Почему отложено:** хорошая идея для будущего, но не заменяет CLI-команду. Пользователь должен иметь возможность запустить проверку в любой момент, не только при коммите. Hook можно добавить позже поверх существующей команды.

## Migration

Миграции не требуется — команды добавляются к существующему каркасу из RFC-0002/RFC-0003.

Шаги после принятия:

1. Добавить `VALID_STATUSES`, `is_valid_transition` и `update_frontmatter_field` в `src/rfclib/rfc.rs`
2. Добавить `rebuild_index` в `src/rfclib/index.rs`
3. Создать `src/commands/set.rs`, `check.rs`, `reindex.rs`
4. Расширить `Commands` в `src/cli.rs`
5. Обновить dispatch в `src/main.rs`
6. Написать тесты (см. план тестирования ниже)
7. Привязать файлы к RFC: `rfc-cli link 0004 src/commands/set.rs src/commands/check.rs ...`
