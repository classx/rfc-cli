## Проблема

RFC-0001 описывает общую структуру RFC-процесса и CLI-инструмент, но сам инструмент ещё не существует. Возникает проблема бутстрапа: для управления RFC нужен `rfc-cli`, а для его реализации нужен принятый RFC.

Команды `init` и `new` — минимальный набор, без которого невозможно:
- создать структуру каталогов для хранения RFC
- создавать новые RFC-документы из шаблона
- начать использовать RFC-процесс на практике

Эти две команды разблокируют весь остальной процесс. До их реализации всё делается вручную.

## Задача

Реализовать команды `rfc-cli init` и `rfc-cli new` как первый шаг к рабочему CLI-инструменту.

### Требования к `init`

1. Команда должна создавать каталог `docs/rfcs/`, если он не существует.
2. Команда должна создавать пустой индексный файл `docs/rfcs/.index.json`, если он не существует.
3. Если каталог и индекс уже существуют — команда должна сообщать об этом и завершаться без ошибки (идемпотентность).
4. Корень проекта определяется по переменной окружения `RFC_HOME`. Если она не задана — используется текущий рабочий каталог.

### Требования к `new`

6. Команда должна принимать название RFC как аргумент: `rfc-cli new "название"`.
7. Команда должна сканировать `docs/rfcs/` и определять следующий свободный номер как максимальный существующий + 1.
8. Если каталог `docs/rfcs/` пуст — первый номер должен быть `0001`.
9. Команда должна генерировать файл `docs/rfcs/NNNN.md` из шаблона со статусом `draft`.
10. Команда должна обновлять индексный файл `.index.json` после создания.
11. Команда должна выводить путь к созданному файлу.
12. Если каталог `docs/rfcs/` не существует — команда должна предлагать выполнить `rfc-cli init`.

### Требования к инфраструктуре

13. Должен быть настроен каркас CLI на базе `clap` с подкомандами.
14. Должен быть реализован парсер YAML frontmatter (чтение текста между `---` маркерами).
15. Должна быть реализована структура данных для индексного файла.
16. Код должен быть организован в модули для дальнейшего расширения.

### Определение CLI (cli.rs)

Используется `clap` с derive API:

```rust
use clap::{Parser, Subcommand};

#[derive(Parser)]
#[command(name = "rfc-cli", about = "Управление RFC-документами")]
pub struct Cli {
    #[command(subcommand)]
    pub command: Commands,
}

#[derive(Subcommand)]
pub enum Commands {
    /// Инициализация структуры RFC в проекте
    Init,
    /// Создание нового RFC из шаблона
    New {
        /// Название RFC
        title: String,
    },
}
```

Структура `Commands` расширяется в будущих RFC новыми подкомандами.

### Определение корня проекта (lib/project.rs)

```rust
pub fn get_project_root() -> Result<PathBuf, Error>
```

Алгоритм:
1. Проверить переменную окружения `RFC_HOME`
2. Если задана — использовать её значение как корень проекта
3. Если не задана — взять текущий рабочий каталог (`std::env::current_dir()`)
4. Вернуть результат как корень проекта

### Команда init (commands/init.rs)

```rust
pub fn execute(project_root: &Path) -> Result<(), Error>
```

Алгоритм:
1. Определить путь: `{project_root}/docs/rfcs/`
2. Если каталог не существует — создать (включая `docs/`, если нужно)
3. Если `.index.json` не существует — создать с пустым содержимым `{"rfcs": []}`
4. Вывести результат:
   - `Created docs/rfcs/` — если создан
   - `Created docs/rfcs/.index.json` — если создан
   - `Already initialized.` — если всё уже на месте

### Команда new (commands/new.rs)

```rust
pub fn execute(project_root: &Path, title: &str) -> Result<(), Error>
```

Алгоритм:
1. Проверить, что `docs/rfcs/` существует. Если нет — вывести: `Error: docs/rfcs/ not found. Run "rfc-cli init" first.`
2. Сканировать `docs/rfcs/*.md`, извлечь номера из имён файлов
3. Определить следующий номер: `max + 1`, или `1` если файлов нет
4. Форматировать номер с ведущими нулями: `format!("{:04}", number)`
5. Сгенерировать содержимое из шаблона:

```
---
title: "RFC-NNNN: {title}"
status: draft
dependencies: []
superseded_by: null
links: []
---
```

## Альтернативы

### 1. Реализовать все команды сразу

Написать весь CLI за один RFC.

**Почему не подходит:** слишком большой скоуп для одного RFC. Невозможно обсуждать и ревьюить детали реализации 13 команд одновременно. Декомпозиция позволяет двигаться итеративно.

### 2. Использовать shell-скрипт для бутстрапа

Написать `init.sh` и `new.sh`, а CLI реализовать позже.

**Почему не подходит:** два инструмента для одной задачи. Скрипты придётся поддерживать параллельно, потом выбрасывать. Проще сразу заложить правильную архитектуру на Rust.

### 3. Начать с другой команды

Например, сначала `list` или `check`.

**Почему не подходит:** без `init` и `new` нечего ни листать, ни проверять. Эти команды — обязательный фундамент.

## Миграция

Это первая реализация — миграции не требуется.

Шаги после принятия:

1. Добавить зависимости в `Cargo.toml`
2. Создать структуру модулей (`src/cli.rs`, `src/commands/`, `src/lib/`)
3. Реализовать `init`
4. Реализовать `new`
5. Написать тесты
6. Привязать файлы к RFC: `rfc-cli link 0002 src/main.rs src/cli.rs ...`
