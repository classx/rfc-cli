## Проблема

После реализации RFC-0002 (`init`, `new`) в проекте можно создавать RFC-документы, но нет возможности:

1. **Просмотреть список RFC.** Чтобы узнать, какие RFC существуют и в каком они статусе, нужно вручную заходить в `docs/rfcs/` и открывать файлы по одному.

2. **Быстро прочитать RFC.** Нет способа вывести содержимое RFC в терминал без навигации по файловой системе.

3. **Узнать статус RFC.** Чтобы проверить статус, нужно открыть файл и найти поле `status` в frontmatter вручную.

4. **Быстро открыть RFC для редактирования.** Нужно помнить путь к файлу и вводить его целиком.

Эти команды — базовый набор для чтения и навигации по RFC. Без них процесс неудобен и люди будут обходить его стороной.

## Задача

Реализовать команды `list`, `view`, `status` и `edit` — набор для чтения и навигации по RFC-документам.

### Требования к `list`

1. Команда должна выводить таблицу всех RFC с колонками: номер, статус, название.
2. Команда должна поддерживать фильтрацию по статусу через флаг `--status <status>`.
3. Команда должна читать данные из индексного файла `.index.json`, не парся RFC-файлы напрямую.
4. Перед чтением индекса команда должна выполнять актуализацию по `mtime` (см. RFC-0001, секция «Индексный файл»).
5. Если RFC не найдено (или ни один не соответствует фильтру) — команда должна выводить сообщение `No RFCs found.`
6. Таблица должна быть отсортирована по номеру RFC по возрастанию.

### Требования к `view`

7. Команда должна принимать номер RFC как аргумент: `rfc-cli view <number>`.
8. Номер можно указывать с ведущими нулями или без: `rfc-cli view 1` и `rfc-cli view 0001` эквивалентны.
9. Команда должна выводить полное содержимое RFC-файла в терминал.
10. Если RFC с указанным номером не существует — команда должна выводить ошибку: `Error: RFC-NNNN not found.`

### Требования к `status`

11. Команда должна принимать номер RFC как аргумент: `rfc-cli status <number>`.
12. Команда должна выводить текущий статус RFC в формате: `RFC-NNNN: <status>`.
13. Команда должна читать данные из индексного файла, не открывая сам RFC-файл.
14. Если RFC с указанным номером не существует — команда должна выводить ошибку.

### Требования к `edit`

15. Команда должна принимать номер RFC как аргумент: `rfc-cli edit <number>`.
16. Команда должна открывать RFC-файл в редакторе, определённом переменной окружения `$EDITOR`.
17. Если `$EDITOR` не задана — команда MUST выводить ошибку: `Error: $EDITOR is not set. Run: export EDITOR=vim`
18. Если RFC с указанным номером не существует — команда должна выводить ошибку.
19. После закрытия редактора команда должна обновить запись в индексе (пересчитать `mtime` и перепарсить frontmatter).
20. Если RFC в статусе `accepted` или `implemented` — команда должна отказывать с ошибкой: `Error: RFC-NNNN is <status>. Use 'rfc-cli view NNNN' to read, or 'rfc-cli edit NNNN --force' to edit anyway.`
21. Флаг `--force` снимает блокировку п.20 — команда открывает редактор с предупреждением: `Warning: RFC-NNNN is <status>. Changes to accepted/implemented RFCs require a new superseding RFC.`

### Вспомогательная функция: нормализация номера

Используется в `view`, `status`, `edit` и последующих командах. Размещается в `src/rfclib/rfc.rs`:

```rust
/// Нормализует номер RFC: "1" → "0001", "0001" → "0001", "42" → "0042"
pub fn normalize_number(input: &str) -> Result<String, String> {
    let n: u32 = input
        .parse()
        .map_err(|_| format!("Invalid RFC number: '{}'. Expected a positive integer.", input))?;
    Ok(format!("{:04}", n))
}

/// Возвращает путь к RFC-файлу по номеру
pub fn rfc_path(project_root: &Path, number: &str) -> Result<PathBuf, String> {
    let normalized = normalize_number(number)?;
    Ok(project_root.join("docs/rfcs").join(format!("{}.md", normalized)))
}
```

### Актуализация индекса

Перед чтением индекса в командах `list` и `status` выполняется актуализация. Функция размещается в `src/rfclib/index.rs`:

```rust
/// Актуализирует индекс: сверяет mtime файлов с записями,
/// перепарсивает изменённые, добавляет новые, удаляет отсутствующие.
pub fn refresh_index(project_root: &Path, index: &mut Index) -> Result<(), Error>
```

Алгоритм:
1. Сканировать `docs/rfcs/*.md`
2. Для каждого файла сравнить `mtime` с записью в индексе
3. Если файл новее и запись в индексе имеет статус `accepted` или `implemented` — пересчитать `content_hash` и сравнить с сохранённым. Если хеш изменился — вернуть ошибку: `Error: RFC-NNNN (accepted) was modified. Changes to accepted/implemented RFCs require a new superseding RFC.` Индекс в этом случае не обновляется.
4. Если файл новее или отсутствует в индексе (и проверка п.3 пройдена) — перепарсить frontmatter и обновить/добавить запись
5. Удалить записи, для которых файл больше не существует
6. Сохранить индекс на диск

### Команда list (commands/list.rs)

```rust
pub fn execute(project_root: &Path, status_filter: Option<&str>) -> Result<(), Error>
```

Алгоритм:
1. Загрузить индекс (`load_index`)
2. Актуализировать (`refresh_index`)
3. Если задан `--status` — отфильтровать записи

Формат вывода:

```
 0002   review         реализация команд init и new
 0003   draft          реализация команд list, view, status и edit
```

Если после фильтрации список пуст — вывести: `No RFCs found.`

### Команда view (commands/view.rs)

```rust
pub fn execute(project_root: &Path, number: &str) -> Result<(), Error>
```

Алгоритм:
1. Нормализовать номер (`normalize_number`)
2. Определить путь к файлу (`rfc_path`)
3. Проверить, что файл существует. Если нет — `Error: RFC-NNNN not found.`
4. Прочитать содержимое файла
5. Вывести в stdout

### Команда status (commands/status.rs)

```rust
pub fn execute(project_root: &Path, number: &str) -> Result<(), Error>
```

Алгоритм:
1. Загрузить и актуализировать индекс
2. Нормализовать номер
3. Найти запись в индексе. Если не найдена — `Error: RFC-NNNN not found.`
4. Вывести: `RFC-NNNN: <status>`

Пример: `rfc-cli status 1` → `RFC-0001: accepted`

### Команда edit (commands/edit.rs)

```rust
pub fn execute(project_root: &Path, number: &str, force: bool) -> Result<(), Error>
```

Алгоритм:
1. Прочитать переменную окружения `$EDITOR`. Если не задана — вывести ошибку с подсказкой
2. Нормализовать номер, определить путь к файлу (абсолютный: `{project_root}/docs/rfcs/NNNN.md`)
3. Проверить, что файл существует
4. Загрузить индекс и проверить статус RFC. Если статус `accepted` или `implemented`:
   - Без `--force` — вернуть ошибку: `Error: RFC-NNNN is <status>. Use 'rfc-cli view NNNN' to read, or 'rfc-cli edit NNNN --force' to edit anyway.`
   - С `--force` — вывести предупреждение: `Warning: RFC-NNNN is <status>. Changes to accepted/implemented RFCs require a new superseding RFC.` и продолжить
5. Запустить процесс: `$EDITOR {абсолютный_путь_к_файлу}` и дождаться завершения
6. После закрытия редактора — актуализировать запись в индексе (пересчитать `mtime` и перепарсить frontmatter)

## Альтернативы

### 1. Вывод view через пейджер (less/more)

Вместо прямого вывода в stdout прогонять содержимое через `$PAGER` или `less`.

**Почему отложено:** добавляет сложность (обработка `$PAGER`, fallback). Для MVP достаточно прямого вывода — пользователь может сам пайпнуть в `less`: `rfc-cli view 1 | less`.

### 2. Команда `show` вместо `view`

Назвать команду `show` по аналогии с `git show`.

**Почему не подходит:** `view` уже зафиксирована в RFC-0001 (`accepted`). Изменение имени команды потребует нового RFC, замещающего RFC-0001. Не оправдано на данном этапе.

### 3. Объединить `status` и `view`

Сделать `status` флагом команды `view`: `rfc-cli view 0001 --status-only`.

**Почему не подходит:** `status` читает только индекс (быстро), `view` читает файл целиком. Разные паттерны использования — удобнее как отдельные команды.

## Миграция

Миграции не требуется — команды добавляются к существующему каркасу из RFC-0002.

Шаги после принятия:

1. Добавить `normalize_number` и `rfc_path` в `src/rfclib/rfc.rs`
2. Реализовать `refresh_index` в `src/rfclib/index.rs`
3. Создать `src/commands/list.rs`, `view.rs`, `status.rs`, `edit.rs`
4. Расширить `Commands` в `src/cli.rs`
5. Написать тесты (см. план тестирования ниже)
6. Привязать файлы к RFC: `rfc-cli link 0003 src/commands/list.rs src/commands/view.rs ...`

### План тестирования

#### Вспомогательная функция

```rust
/// Запуск rfc-cli с кастомным $EDITOR
fn run_rfc_cli_with_editor(project_dir: &Path, args: &[&str], editor: &str) -> std::process::Output {
    let binary = env!("CARGO_BIN_EXE_rfc-cli");
    std::process::Command::new(binary)
        .args(args)
        .env("RFC_HOME", project_dir.as_os_str())
        .env("EDITOR", editor)
        .output()
        .expect("Failed to execute rfc-cli")
}
```

#### Тесты для `list`

| Тест | Что проверяем |
|---|---|
| `list` при пустом индексе | Вывод `No RFCs found.` |
| `list` с несколькими RFC | Таблица содержит все RFC, отсортированные по номеру |
| `list --status draft` | Только RFC со статусом `draft` |
| `list --status accepted` при отсутствии таких | `No RFCs found.` |

#### Тесты для `view`

| Тест | Что проверяем |
|---|---|
| `view 1` | stdout содержит полный контент `0001.md` |
| `view 0001` | эквивалентно `view 1` (нормализация) |
| `view 99` | Ошибка `RFC-0099 not found.` |
| `view abc` | Ошибка невалидного номера |

#### Тесты для `status`

| Тест | Что проверяем |
|---|---|
| `status 1` | Вывод `RFC-0001: draft` |
| `status 0001` | эквивалентно `status 1` |
| `status 99` | Ошибка `RFC-0099 not found.` |

#### Тесты для `edit`

Команда `edit` запускает внешний процесс `$EDITOR`. Для тестирования `$EDITOR` подменяется на скрипт/команду с предсказуемым поведением.

Типичные подстановки:
- `true` — «редактор», который сразу завершается без изменений
- `sed -i 's/old/new/'` — «редактор», который меняет содержимое файла

| Тест | `$EDITOR` | Что проверяем |
|---|---|---|
| Нет `$EDITOR` (`env_remove("EDITOR")`) | убран | Ошибка `$EDITOR is not set` |
| RFC не существует | `true` | Ошибка `RFC-0099 not found.` |
| accepted RFC без `--force` | `true` | Блокировка с подсказкой про `--force` |
| accepted RFC с `--force` | `true` | Warning + успешное завершение |
| Обновление индекса после редактирования | `sed -i 's/original/changed/'` | title в `.index.json` обновился |
| Нормализация номера | `true` | `edit 1` эквивалентно `edit 0001` |

**Примечание:** `sed -i` на macOS требует `sed -i ''` (с пустым аргументом). В CI (ubuntu-latest) работает без него. Для кроссплатформенности можно использовать `sh -c "..."` обёртку.
