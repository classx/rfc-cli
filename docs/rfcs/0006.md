## Проблема

После реализации RFC-0002 — RFC-0005 CLI покрывает полный цикл управления RFC-документами: создание, просмотр, редактирование, управление статусом, валидацию формата, связь с кодом и навигацию по зависимостям. Реализовано 12 из 13 команд, определённых в RFC-0001.

Последняя нереализованная команда — `doctor`. В отличие от `check`, который валидирует **формат** (frontmatter, обязательные секции, content_hash), `doctor` анализирует **смысловую согласованность** между RFC и кодовой базой.

Без `doctor` остаются незакрытыми следующие проблемы:

1. **Дрифт кода.** RFC принят, но файлы из `links` изменились после принятия — реализация ушла от спецификации. Разработчик не узнает об этом, пока вручную не сравнит даты.

2. **Забытая реализация.** RFC в статусе `accepted` не имеет ни одной ссылки в `links` — возможно, реализацию забыли привязать или вовсе не начали.

3. **Мёртвые ссылки.** Файлы из `links` удалены или переименованы. `check` уже обнаруживает это, но `doctor` даёт контекст: когда файл пропал, какой RFC затронут, что делать.

4. **Зависшие черновики.** RFC в статусе `draft` не обновлялся длительное время — возможно, заброшен и засоряет список.

5. **Незакрытые зависимости.** RFC в статусе `accepted` зависит от RFC, который ещё не `accepted`/`implemented` — нельзя корректно реализовать зависимый RFC.

6. **Циклические зависимости.** Граф зависимостей содержит цикл — потенциальная логическая ошибка в структуре RFC.

Команда `doctor` замыкает спецификацию RFC-0001 и делает CLI полноценным инструментом для поддержания здоровья RFC-процесса в проекте.

## Задача

Реализовать команду `rfc-cli doctor` — диагностику здоровья связи между RFC и кодовой базой.

### Требования

1. Команда запускается без аргументов: `rfc-cli doctor`.
2. Команда проходит по всем RFC в индексе и выполняет 6 диагностических проверок.
3. Результаты группируются по RFC: сначала номер и название, затем список проблем.
4. RFC без проблем не выводятся (тихий отчёт — только проблемы).
5. После списка проблем выводится итоговая строка.
6. Код выхода: `0` если нет ошибок (❌), `1` если есть хотя бы одна ошибка. Предупреждения (⚠️) не влияют на код выхода.
7. Опционально: флаг `--stale-days <N>` для настройки порога «зависшего черновика» (по умолчанию 30 дней).

### Диагностические проверки

| # | Проверка | Уровень | Условие |
|---|----------|---------|---------|
| D1 | Дрифт кода | ❌ ошибка | RFC `accepted`/`implemented`, файл из `links` имеет `mtime` новее, чем `mtime` RFC-файла |
| D2 | Нет реализации | ⚠️ предупреждение | RFC `accepted` и `links` пуст |
| D3 | Мёртвые ссылки | ❌ ошибка | Файл из `links` не существует на диске |
| D4 | Зависший черновик | ⚠️ предупреждение | RFC `draft`, `mtime` старше `--stale-days` дней |
| D5 | Незакрытые зависимости | ⚠️ предупреждение | RFC `accepted`, зависит от RFC не в статусе `accepted`/`implemented` |
| D6 | Циклические зависимости | ❌ ошибка | Обнаружен цикл в графе зависимостей |

### Формат вывода

```
RFC-0003 (cache):
  ❌ code drift: src/cache/mod.rs modified after RFC acceptance
  ❌ dead link: src/cache/old_store.rs (file not found)

RFC-0005 (logging):
  ⚠️  no linked files (status: accepted)

RFC-0008 (auth):
  ⚠️  stale draft (last modified 45 days ago)

RFC-0012 (api-v2):
  ⚠️  depends on RFC-0009 (metrics) which is still in review

Summary: 3 error(s), 3 warning(s) across 4 RFC(s).
```

Если всё чисто:

```
All RFCs are healthy. ✅
```

## Дизайн

### Новый модуль

```
src/commands/doctor.rs  — команда doctor
```

### Структура диагностики

Каждая проверка — отдельная функция, возвращающая вектор диагностик. Это упрощает тестирование и добавление новых проверок в будущем.

```rust
/// Severity level of a diagnostic
#[derive(Debug, Clone, PartialEq)]
pub enum Severity {
    Error,   // ❌
    Warning, // ⚠️
}

/// Single diagnostic finding
#[derive(Debug, Clone)]
pub struct Diagnostic {
    pub rfc_number: String,
    pub severity: Severity,
    pub message: String,
}
```

### Расширение CLI (cli.rs)

```rust
pub enum Commands {
    // ... существующие команды ...

    /// Диагностика здоровья RFC-проекта
    Doctor {
        /// Порог дней для определения зависшего черновика
        #[arg(long, default_value = "30")]
        stale_days: u64,
    },
}
```

### Команда doctor (commands/doctor.rs)

```rust
pub fn execute(project_root: &Path, stale_days: u64) -> Result<(), String>
```

Алгоритм:

1. Загрузить и обновить индекс (`load_index`, `refresh_index`).
2. Для каждой записи в индексе выполнить все 6 проверок, собирая `Vec<Diagnostic>`.
3. Дополнительно: запустить проверку циклических зависимостей на графе всех RFC.
4. Сгруппировать диагностики по `rfc_number`.
5. Вывести отчёт.
6. Вернуть ошибку (exit code 1), если есть хотя бы одна диагностика с `Severity::Error`.

### Реализация проверок

#### D1: Дрифт кода

```rust
fn check_code_drift(
    project_root: &Path,
    entry: &IndexEntry,
) -> Vec<Diagnostic>
```

Для RFC в статусе `accepted`/`implemented`:
- Получить `mtime` RFC-файла из индекса (или с диска).
- Для каждого пути в `links` получить `mtime` файла.
- Если `mtime` файла > `mtime` RFC → добавить ❌ диагностику.

Примечание: сравниваем `mtime` файлов с диска, а не из индекса, чтобы получить актуальные данные.

#### D2: Нет реализации

```rust
fn check_no_implementation(entry: &IndexEntry) -> Vec<Diagnostic>
```

Если `entry.status == "accepted"` и `entry.links.is_empty()` → ⚠️.

Не применяется к `implemented` — предполагаем, что если статус `implemented`, ссылки могли быть добавлены и удалены осознанно.

#### D3: Мёртвые ссылки

```rust
fn check_dead_links(
    project_root: &Path,
    entry: &IndexEntry,
) -> Vec<Diagnostic>
```

Для каждого пути в `links`: если `project_root.join(link).exists()` == false → ❌.

Примечание: `check` уже делает эту проверку, но `doctor` включает её для полноты отчёта, чтобы пользователю не нужно было запускать обе команды.

#### D4: Зависший черновик

```rust
fn check_stale_draft(
    entry: &IndexEntry,
    stale_days: u64,
) -> Vec<Diagnostic>
```

Если `entry.status == "draft"`:
- Распарсить `entry.mtime` как Unix timestamp (секунды).
- Вычислить разницу с текущим временем.
- Если разница > `stale_days * 86400` → ⚠️ с указанием количества дней.

#### D5: Незакрытые зависимости

```rust
fn check_unresolved_dependencies(
    entry: &IndexEntry,
    index: &Index,
) -> Vec<Diagnostic>
```

Если `entry.status == "accepted"`:
- Для каждого элемента `dependencies` найти соответствующую запись в индексе.
- Если зависимость не в статусе `accepted`/`implemented` → ⚠️.

#### D6: Циклические зависимости

```rust
fn check_dependency_cycles(index: &Index) -> Vec<Diagnostic>
```

Алгоритм обнаружения циклов — DFS с тремя состояниями (white/gray/black):

1. Построить граф: для каждого RFC номер → список зависимостей (номера).
2. Для каждого узла запустить DFS.
3. Если при обходе встречается gray-узел (уже в стеке) → цикл обнаружен.
4. Вернуть ❌ диагностику для каждого участника цикла.

Для предсказуемости: обходить узлы в порядке возрастания номеров.

### Взаимодействие с существующими командами

- **`check`** — валидирует формат. `doctor` — валидирует семантику. Они дополняют друг друга. Пересечение: мёртвые ссылки (D3) проверяются обоими, но `doctor` даёт общую картину.
- **`deps`** — показывает дерево зависимостей одного RFC. `doctor` D5/D6 — анализирует зависимости всех RFC сразу.
- **`link`/`unlink`** — управляют ссылками. `doctor` D1/D2/D3 — проверяют корректность этих ссылок.

### Расширение main.rs

```rust
Commands::Doctor { stale_days } => commands::doctor::execute(&project_root, stale_days),
```

## Альтернативы

### 1. Объединить `doctor` и `check` в одну команду

Можно добавить флаг `--deep` или `--semantic` к `check` вместо отдельной команды.

**Отклонено**: RFC-0001 определяет их как отдельные команды с разными ответственностями. `check` — быстрая формальная проверка для CI. `doctor` — глубокая диагностика для разработчика. Разная аудитория, разная скорость выполнения.

### 2. `doctor` только для одного RFC

Можно добавить аргумент `rfc-cli doctor [<number>]` для проверки конкретного RFC.

**Отклонено на данном этапе**: `doctor` по определению анализирует состояние проекта в целом. Проверка циклов (D6) и незакрытых зависимостей (D5) имеет смысл только на полном графе. Можно добавить фильтр позже, если отчёт станет слишком длинным.

### 3. Машиночитаемый вывод (JSON)

Можно добавить `--format json` для интеграции с другими инструментами.

**Отложено**: на текущем этапе достаточно человекочитаемого вывода. JSON-формат можно добавить в следующей итерации, когда появятся интеграции.

### 4. Без проверки циклов (D6)

Циклические зависимости маловероятны в небольших проектах, реализация DFS добавляет сложности.

**Отклонено**: реализация DFS тривиальна (~30 строк). Обнаружение циклов на раннем этапе предотвращает трудноуловимые проблемы в будущем. Это единственная проверка, которую невозможно сделать иначе.

### 5. Порог `stale-days` в конфиге, а не в аргументе

Можно хранить конфигурацию `doctor` в файле (например, `.rfc-cli.toml`).

**Отложено**: для одного параметра файл конфигурации избыточен. CLI-аргумент с дефолтом достаточен. Конфиг можно добавить, когда появятся другие настраиваемые параметры.

## Миграция

Изменения полностью обратно совместимы. Добавляется одна новая команда, существующие команды не меняются.

### Шаги реализации

1. Создать `src/commands/doctor.rs` с типами `Severity`, `Diagnostic` и функциями проверок D1–D6.
2. Реализовать `execute()` — оркестрация проверок, группировка вывода, определение exit code.
3. Расширить `src/cli.rs` — добавить вариант `Doctor` с `--stale-days`.
4. Расширить `src/main.rs` — маршрутизация.
5. Зарегистрировать модуль в `src/commands/mod.rs`.
6. Написать интеграционные тесты.

### План тестирования

#### Тесты для D1 (дрифт кода)

1. `test_doctor_code_drift` — файл из `links` новее RFC → ❌ в выводе.
2. `test_doctor_no_drift` — файл из `links` старше RFC → чисто.

#### Тесты для D2 (нет реализации)

3. `test_doctor_no_implementation` — `accepted` RFC без `links` → ⚠️.
4. `test_doctor_accepted_with_links` — `accepted` RFC с `links` → чисто.

#### Тесты для D3 (мёртвые ссылки)

5. `test_doctor_dead_link` — файл удалён → ❌.

#### Тесты для D4 (зависший черновик)

6. `test_doctor_stale_draft` — `draft` RFC с древним `mtime` → ⚠️.
7. `test_doctor_fresh_draft` — `draft` RFC с недавним `mtime` → чисто.
8. `test_doctor_stale_days_flag` — `--stale-days 0` → любой draft считается stale.

#### Тесты для D5 (незакрытые зависимости)

9. `test_doctor_unresolved_deps` — `accepted` RFC зависит от `draft` → ⚠️.
10. `test_doctor_resolved_deps` — `accepted` RFC зависит от `accepted` → чисто.

#### Тесты для D6 (циклические зависимости)

11. `test_doctor_cycle_detected` — A→B→A → ❌.
12. `test_doctor_no_cycle` — линейные зависимости → чисто.

#### Общие тесты

13. `test_doctor_healthy_project` — чистый проект → «All RFCs are healthy. ✅».
14. `test_doctor_exit_code_error` — есть ❌ → exit code 1.
15. `test_doctor_exit_code_warning_only` — только ⚠️ → exit code 0.
16. `test_doctor_empty_project` — нет RFC → «All RFCs are healthy. ✅».
