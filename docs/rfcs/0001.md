## Problem

Разработка с использованием AI-ассистентов страдает от отсутствия формальных спецификаций:

1. **Нет единого источника правды.** Проектные решения принимаются в чатах, в голове или на лету. Они нигде не зафиксированы, и через неделю никто не помнит, почему выбрали именно такой подход.

2. **AI работает вслепую.** Без чёткого описания целевой архитектуры, форматов данных и ограничений AI-ассистент опирается на собственные предположения. Результат — код, который нужно переделывать.

3. **Решения не проходят ревью до реализации.** Код пишется раньше, чем идея обсуждена и проверена. Ошибки в дизайне обнаруживаются слишком поздно.

4. **Нет истории решений.** Когда возникает вопрос «почему сделано именно так?», ответа нет. Альтернативы, которые рассматривались и были отвергнуты, не задокументированы.

5. **Потеря контекста между сессиями.** Каждый новый чат с AI начинается с чистого листа. Без документа, который можно передать в контекст, приходится заново объяснять архитектуру, ограничения и принятые решения.

6. **Нет критериев готовности.** Без спецификации непонятно, когда фича «сделана». Нет формального определения done — ни для человека, ни для AI.

7. **Размытие границ задачи.** Без фиксированного скоупа AI генерирует больше или меньше, чем нужно. Задача обрастает неожиданными деталями или, наоборот, теряет важные.

8. **Невоспроизводимость результата.** Один и тот же запрос к AI в разные дни даёт разный код. RFC фиксирует ожидаемое поведение, делая результат проверяемым.

9. **Сложность онбординга.** Новый участник (или ты сам через месяц) не может понять, почему система устроена именно так, без раскопок в git log и чатах.

RFC (Request for Comments) — формат, решающий эти проблемы. Это текстовые документы, в которых:
- строго описано поведение системы
- приведены протоколы, форматы данных и интерфейсы
- зафиксированы требования с уровнями обязательности (MUST, SHOULD, MAY)
- описаны альтернативы и обоснование выбранного решения

## Goal

Внедрить RFC как обязательный этап разработки: идея сначала описывается, обсуждается и утверждается — и только потом реализуется.

### Document Format

1. Каждый RFC должен храниться в `docs/rfcs/` в формате Markdown.
2. Имя файла должно соответствовать шаблону `NNNN.md`, где `NNNN` — порядковый номер с ведущими нулями (например, `0001.md`).
3. Документ должен содержать YAML frontmatter с полями:
   - `title` — название RFC
   - `status` — текущий статус
   - `dependencies` — список зависимостей от других RFC (может быть пустым)
   - `superseded_by` — номер RFC, который заменяет текущий (например, `RFC-0005`), или `null`
   - `links` — список путей к файлам исходного кода, связанным с этим RFC (может быть пустым)
4. Документ должен содержать секции: **Проблема**, **Задача**, **Дизайн**, **Альтернативы**.
5. Документ может содержать секции: **Голосование**, **Миграция**.

### Lifecycle

6. Каждый RFC должен проходить через статусы в следующем порядке:
   - `draft` — черновик, идёт написание
   - `review` — документ готов к обсуждению
   - `accepted` — решение утверждено, можно реализовывать
   - `implemented` — реализация завершена
   - `superseded` — заменён новым RFC (поле `superseded_by` указывает на замену)
   - `deprecated` — RFC устарел или отменён без замены
7. Реализация не должна начинаться до перехода RFC в статус `accepted`.
8. Переход из `draft` в `review` может происходить только когда заполнены все обязательные секции.
9. Изменить или отменить принятый RFC можно только через новый RFC. Прямое редактирование документа в статусе `accepted` или `implemented` запрещено. Новый RFC при принятии переводит старый в статус `superseded` и заполняет поле `superseded_by`.

### CLI Tool

9. Необходимо реализовать CLI-инструмент `rfc-cli` для управления RFC.
10. Инструмент должен уметь:
11. - создать файловую структуру (init)
    - создавать новый RFC из шаблона с автоматической нумерацией (new)
    - выводить список RFC с фильтрацией по статусу (list)
    - валидировать формат RFC (frontmatter, обязательные секции) (check)
    - открывать RFC по номеру для просмотра (view)
    - открывать RFC по номеру для редактирования в редакторе (edit)
    - показывать статус по RFC по номеру
    - связывать RFC с файлами исходного кода (link)
    - показывать зависимости RFC (deps)
    - полностью перестраивать индекс (reindex)
    - диагностировать здоровье связи между RFC и кодом (doctor)
    - менять статус RFC с проверкой допустимых переходов (set)
11. Инструмент может уметь:

### Process

12. Любое нетривиальное изменение в архитектуре, API или процессах ЗАПРЕЩЕНО без RFC. Код не пишется до тех пор, пока RFC не перейдёт в статус `accepted`.
13. RFC должен быть доступен AI-ассистенту как контекст при реализации описанной в нём задачи.
14. Отвергнутые альтернативы могут быть задокументированы в секции **Альтернативы**, чтобы не возвращаться к ним повторно.

## Design

### Project Structure

```
docs/
  rfcs/
    .index.json
    0001.md
    0002.md
    ...
```

Все RFC хранятся в каталоге `docs/rfcs/` относительно корня проекта. Команда `rfc-cli init` создаёт этот каталог и пустой индексный файл `.index.json`, если они не существуют.

### Index File

Для производительности на больших проектах (сотни RFC) метаданные кэшируются в файле `docs/rfcs/.index.json`. Формат:

```
{
  "rfcs": [
    {
      "number": "0001",
      "title": "структура RFC",
      "status": "draft",
      "dependencies": [],
      "superseded_by": null,
      "links": [],
      "mtime": "2025-01-15T12:34:56Z",
      "content_hash": null
    }
  ]
}
```

**Стратегия актуализации:**

- При каждом вызове CLI сравнивается `mtime` (время модификации) каждого файла в `docs/rfcs/` с `mtime`, записанным в индексе
- Если файл новее — его frontmatter перепарсивается и запись в индексе обновляется
- Если файл отсутствует в индексе (новый) — добавляется
- Если запись в индексе ссылается на несуществующий файл — удаляется
- Таким образом, ручное редактирование RFC в любом редакторе не ломает индекс

**Защита целостности принятых RFC:**

- При переходе RFC в статус `accepted` или `implemented` CLI вычисляет SHA-256 хэш содержимого файла и сохраняет его в поле `content_hash` в индексе
- Для RFC в статусах `draft` и `review` поле `content_hash` равно `null` — они свободно редактируются
- При обнаружении изменения `mtime` у файла со статусом `accepted`/`implemented` CLI сравнивает текущий хэш с сохранённым. Если хэши не совпадают — выводит ошибку: `RFC-NNNN has been modified after acceptance. Changes to accepted RFCs are forbidden — create a new RFC instead.`
- Переход в `superseded` или `deprecated` обновляет `content_hash`, так как это легитимное изменение через CLI

Команды `list`, `deps`, `status` читают только индекс, не открывая сами RFC-файлы. Команды `view`, `edit`, `check` обращаются к файлам напрямую.

Файл `.index.json` SHOULD быть добавлен в `.gitignore` — он генерируемый и воспроизводимый.

### RFC Document Template

При создании нового RFC (`rfc-cli new "название"`) генерируется файл со следующим содержимым:

```
---
title: "RFC-NNNN: название"
status: draft
dependencies: []
superseded_by: null
links: []
---

## Problem

## Goal

## Design

## Alternatives

## Voting

## Migration
```

Номер `NNNN` определяется автоматически как максимальный существующий номер + 1.

### YAML Frontmatter Schema

| Поле           | Тип        | Обязательное | Описание                                      |
|----------------|------------|--------------|-----------------------------------------------|
| `title`        | string     | да           | Название в формате `RFC-NNNN: краткое описание` |
| `status`       | string     | да           | Один из: `draft`, `review`, `accepted`, `implemented`, `superseded`, `deprecated` |
| `dependencies` | list       | да           | Список номеров RFC-зависимостей, например `[RFC-0001]`. Пустой список `[]`, если зависимостей нет |
| `superseded_by`| string/null| да           | Номер RFC-замены (например, `RFC-0005`) или `null`, если не заменён |
| `links`        | list       | да           | Список путей к файлам исходного кода, связанным с RFC. Пустой список `[]`, если связей нет |

### Valid Status Transitions

```
draft → review → accepted → implemented
                    ↓            ↓
               deprecated    superseded
                                ↑
               deprecated ← accepted

draft → deprecated (отмена на ранней стадии)
```

Любые другие переходы запрещены. При попытке недопустимого перехода CLI выдаёт ошибку.

При переходе в `superseded` CLI автоматически заполняет поле `superseded_by` и проверяет, что замещающий RFC существует.

Все переходы статусов выполняются только через команду `rfc-cli set`. Ручное редактирование поля `status` в файле не рекомендуется — CLI не сможет зафиксировать `content_hash` и обновить индекс.

### CLI Commands

#### `rfc-cli init`

Создаёт каталог `docs/rfcs/` и пустой индексный файл `.index.json`, если они не существуют. Если каталог уже есть — сообщает об этом и завершается без ошибки.

#### `rfc-cli new <title>`

Создаёт новый RFC-файл из шаблона.

- Сканирует `docs/rfcs/` и определяет следующий свободный номер
- Генерирует файл `docs/rfcs/NNNN.md` со статусом `draft`
- Выводит путь к созданному файлу

Пример: `rfc-cli new "парсер frontmatter"` → создаёт `docs/rfcs/0002.md`

#### `rfc-cli list [--status <status>]`

Выводит таблицу всех RFC:

```
 #     Статус        Название
 0001  draft         структура RFC
 0002  accepted      парсер frontmatter
```

При указании `--status draft` — фильтрует по статусу.

#### `rfc-cli view <number>`

Выводит содержимое RFC в терминал. Номер можно указывать с нулями или без: `rfc-cli view 1` и `rfc-cli view 0001` эквивалентны.

#### `rfc-cli edit <number>`

Открывает RFC в редакторе, определённом переменной окружения `$EDITOR`. Если `$EDITOR` не задан — выводит ошибку с подсказкой.

#### `rfc-cli status <number>`

Выводит текущий статус указанного RFC.

#### `rfc-cli set <number> <status>`

Меняет статус RFC с проверкой допустимых переходов. Атомарно выполняет:

1. Проверяет, что переход из текущего статуса в новый разрешён
2. Обновляет поле `status` в YAML frontmatter файла
3. При переходе в `accepted` или `implemented` — вычисляет SHA-256 и сохраняет `content_hash` в индексе
4. При переходе в `superseded` — требует указать замещающий RFC: `rfc-cli set 0003 superseded --by 0010`
5. Обновляет индексный файл

Пример:

```
$ rfc-cli set 0001 accepted
RFC-0001 (структура RFC): review → accepted ✅

$ rfc-cli set 0001 draft
Error: transition accepted → draft is not allowed.

$ rfc-cli set 0003 superseded --by 0010
RFC-0003 (cache): implemented → superseded (by RFC-0010) ✅
```

#### `rfc-cli link <number> <path>`

Связывает RFC с файлом исходного кода. Добавляет путь в поле `links` во frontmatter.

- Проверяет, что указанный файл существует
- Не добавляет дубликаты — если путь уже в `links`, сообщает об этом
- Путь сохраняется относительно корня проекта

Пример: `rfc-cli link 0001 src/lib/example.rs` → добавляет `src/lib/example.rs` в `links` RFC-0001.

#### `rfc-cli unlink <number> <path>`

Удаляет связь RFC с файлом. Убирает путь из поля `links`. Если путь не найден в `links` — выводит ошибку.

#### `rfc-cli deps <number>`

Выводит дерево зависимостей указанного RFC. Читает поле `dependencies` и подтягивает `title` каждого связанного RFC.

Пример: `rfc-cli deps 0007`

```
RFC-0007 depends on:
  - 0003 (config)
  - 0005 (logging)
```

Если зависимостей нет — выводит `RFC-0007 has no dependencies.`

Если зависимость ссылается на несуществующий RFC — выводит предупреждение.

#### `rfc-cli check [<number>]`

Валидирует RFC. Без аргумента — проверяет все. Проверки:

- YAML frontmatter парсится без ошибок
- Все обязательные поля frontmatter присутствуют
- Значение `status` входит в список допустимых
- Обязательные секции (Проблема, Задача, Дизайн, Альтернативы) присутствуют
- Номер в имени файла совпадает с номером в `title`
- Зависимости ссылаются на существующие RFC
- Файлы в `links` существуют в проекте
- RFC в статусах `accepted`/`implemented` не были изменены (сравнение SHA-256 с `content_hash` в индексе)

Результат: список ошибок или сообщение «OK». После проверки индекс перестраивается.

#### `rfc-cli reindex`

Полностью перестраивает индексный файл `docs/rfcs/.index.json` с нуля — сканирует все RFC-файлы и парсит их frontmatter. Используется как страховка, если индекс повреждён или рассинхронизирован.

Пример: `rfc-cli reindex` → `Reindexed 42 RFCs.`

#### `rfc-cli doctor`

Диагностика здоровья связи между RFC и кодовой базой. В отличие от `check` (валидация формата), `doctor` анализирует смысловую согласованность проекта.

Проверки:

- **Код изменился после принятия RFC.** Для RFC в статусе `accepted`/`implemented` проверяются файлы из `links` — если их `mtime` новее, чем у RFC, выводится предупреждение о возможном дрифте.
- **Нет реализации.** RFC в статусе `accepted` без единого файла в `links` — возможно, забыли реализовать или привязать код.
- **Мёртвые ссылки.** Файлы в `links` удалены или переименованы.
- **Зависнувшие черновики.** RFC в статусе `draft` старше заданного порога (по `mtime`) — возможно, заброшены.
- **Незакрытые зависимости.** RFC в статусе `accepted`, зависящий от RFC, который ещё не `accepted`/`implemented`.

Пример вывода:

```
RFC-0003 (cache):
  ❌ code changed but RFC not updated
     src/cache/mod.rs modified 2025-01-20 (RFC accepted 2025-01-10)
  ❌ dead link: src/cache/old_store.rs (file not found)

RFC-0005 (logging):
  ⚠️  no implementation found (status: accepted, links: empty)

RFC-0008 (auth):
  ⚠️  stale draft (last modified 45 days ago)

RFC-0012 (api-v2):
  ⚠️  depends on RFC-0009 (metrics) which is still in review

All other RFCs are healthy. ✅
```

Код выхода: `0` если всё чисто, `1` если есть ошибки (❌), `0` если только предупреждения (⚠️).

### Technology

- Язык: Rust (edition 2021)
- Парсинг YAML frontmatter: выделение текста между первой и второй строкой `---`, десериализация через `serde_yaml`
- Парсинг Markdown-секций: поиск строк, начинающихся с `## `
- CLI-фреймворк: `clap`

## Alternatives

### 1. ADR (Architecture Decision Records)

Формат, близкий к RFC: короткие документы, фиксирующие одно архитектурное решение. Каждый ADR содержит контекст, решение и последствия.

**Почему не подходит:** ADR фиксируют уже принятое решение постфактум. В нашем процессе важен этап обсуждения *до* реализации — с секциями голосования, зависимостей и миграции. Кроме того, ADR не предполагают детальной спецификации поведения системы, а скорее отвечают на вопрос «почему».

### 2. GitHub Issues / Pull Requests

Описание задачи и дизайна прямо в issue или в описании PR.

**Почему не подходит:** Issues заточены под трекинг задач, а не под спецификации. Они плохо структурированы, теряются в потоке, не имеют обязательных секций и валидации. PR-описания привязаны к конкретному коду и не живут отдельно как источник правды. Передать issue как контекст AI-ассистенту возможно, но формат слишком свободный.

### 3. Notion / Google Docs / Wiki

Дизайн-документы в облачном редакторе с совместным доступом.

**Почему не подходит:** Документы живут вне репозитория — нет версионирования через git, нет code review, нет связи с кодом. Ссылки ломаются, доступ зависит от аккаунта. AI-ассистент не может прочитать документ из Notion напрямую, а копирование контента туда-обратно создаёт рассинхронизацию.

### 4. Без формального процесса (статус-кво)

Решения принимаются в чатах, в голове или «по ходу дела».

**Почему не подходит:** Именно эту проблему мы решаем. См. секцию «Проблема».

## Voting

RFC-0001 является основополагающим документом проекта и не требует формального голосования. Принимается по факту начала использования RFC-процесса.

Для последующих RFC процедура голосования определяется следующим образом:

- Голосование проводится при переходе из статуса `review` в `accepted`
- Участники оставляют свою позицию: **за**, **против** (с обоснованием) или **воздержался**
- RFC считается принятым, если нет голосов «против» без разрешения
- Голоса фиксируются в этой секции в формате:
  - `за — @участник, дата, комментарий (опционально)`
  - `против — @участник, дата, причина`

## Migration

Это первый RFC в проекте — миграции с предыдущего процесса не требуется.

Шаги внедрения:

1. Зафиксировать данный RFC (`0001.md`) в репозитории
2. Реализовать минимальный набор команд `rfc-cli`: `init`, `new`, `list`, `check`
3. Все новые проектные решения оформлять как RFC, начиная с момента принятия этого документа
4. Существующие решения, принятые до внедрения RFC, документировать по мере необходимости — обратная сила не требуется
